# 后续更新建议

这份文档用于记录项目后续演进方向。`TODO.md` 更偏任务清单；本文更偏“为什么做、先做什么、做到什么程度算有效”。

## 优先级建议

### P0：保证评分可信

目标：用户上传明显无关音频、英语听力、普通说话、音乐时，不能获得高分；真实模仿哈气时，分数要有可解释的梯度。

建议：

* 建立一组固定测试样本：高相似哈气、中等哈气、低相似哈气、普通说话、英语听力、音乐、纯噪声、静音。
* 每次调整算法后跑同一批样本，记录分数变化，避免“修一个误判，引入另一个误伤”。
* 保留当前 `voicedRatio` 连续人声限制，但继续用真实样本调阈值。
* 把评分理由和扣分上限绑定，让用户看到的理由能对应实际限制条件。

验收标准：

* 英语听力、朗读、普通说话稳定低于 `42` 分。
* 音乐、环境噪声、静音稳定低于 `45` 分。
* 真实模仿哈气可以根据节奏、音色、时长差异拉开分数。

## P1：提升结算页传播性

目标：检测结果更像一张可分享的“结算单”，用户愿意保存或发给朋友。

建议：

* 分享图加入项目二维码或访问链接，方便二次传播。
* 分享图文案使用评分理由，不再使用机械化指标条。
* 为不同分数段准备不同标题，例如“猫猫频道认证”“哈气修炼中”“本次还没接入猫猫频段”。
* 微信无法直接网页静默分享图片，因此保留“保存图片 + 复制文案”的路径，并在按钮文案里说明。

验收标准：

* 手机端分享图清晰，文字不溢出。
* 微信/QQ 场景下至少能顺畅保存图片并复制文案。
* 分享图中能看出分数、称号、理由、项目入口。

## P1：完善移动端录音体验

目标：用户在微信、QQ、抖音内置浏览器和 Safari 中能清楚知道当前能否录音、为什么失败、如何继续。

建议：

* 对麦克风权限拒绝、浏览器不支持、录音为空、上传失败分别给出不同提示。
* 增加上传文件作为录音失败时的替代路径。
* 检测中禁止重复提交，避免同一音频并发请求。
* 对超长音频提前提示“只分析前若干秒”，减少用户误解。

验收标准：

* 权限失败时不出现空白结果页。
* 后端失败时不生成假分数。
* 用户至少能通过上传音频完成一次检测。

## P2：算法升级路线

当前算法是轻量级启发式方案，适合 Demo 展示，但不是专业音频识别。后续可以分阶段升级。

阶段 1：强化传统特征

* 增加 MFCC、谱质心、谱通量、谱平坦度。
* 分离“气声类声音”和“有声语言类声音”。
* 为每个特征记录贡献和限制原因。

阶段 2：建立小型评测集

* 收集 30-50 条真实哈气样本。
* 收集 30-50 条负样本：说话、英语听力、音乐、噪声、动物叫声。
* 固定每次算法改动后的分数对比表。

阶段 3：引入嵌入模型

* 如部署条件允许，可尝试音频 embedding 或轻量分类器。
* 保留启发式硬上限作为安全阀，避免模型偶发高分。

## P2：工程化与部署

目标：让项目更容易演示、部署和继续开发。

建议：

* 为 `/api/upload` 增加文件大小限制、格式检查和临时文件清理策略。
* 增加基础端到端测试，覆盖上传、后端失败、结算页、分享图、本机排行榜。
* 增加部署说明，例如本地 Python 服务、云服务器、反向代理、静态资源路径。
* 如果继续扩展 UI，可迁移到 Vite/React/TypeScript；当前静态实现适合短期 Demo。

## 不建议近期投入

* 不建议立即做复杂账号系统。当前项目更适合轻量传播，账号会增加大量非核心成本。
* 不建议把排行榜直接做成全局公开榜，除非先解决刷榜、隐私和音频存储策略。
* 不建议在没有样本集前频繁微调阈值，否则很容易凭感觉修坏算法。
* 不建议重新加入前端 Mock 打分。当前产品要求是后端实测，Mock 会破坏评分可信度。

## 推荐下一步

1. 先整理真实测试样本目录。
2. 写一个批量评分脚本，输出样本名、分数、称号、命中限制原因。
3. 用这份表调整连续人声、音乐、噪声和真实哈气的阈值。
4. 再优化分享图和移动端录音提示。
